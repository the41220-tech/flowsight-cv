# Cycle 17b — 데이터 증량 재평가 보고서 (MVDet 4뷰, 40→400 프레임)

날짜 2026-06-25 · WILDTRACK 4뷰(C1,C2,C4,C5) · BEV occupancy + peak-NMS(1.0m) · @2m · best-F1
n=40(20 train/20 test, Cycle17) → **n=200(100 train/100 test, 본 사이클, 뷰당 400프레임 추출)**

## 0. 무엇을/왜
Cycle17 결론 = "20프레임 regime은 분산이 커 가설 판정이 신뢰 어렵고, 진짜 천장은 데이터양". 이를 검증하려 EPFL 6.8GB zip을 재다운로드 → 스트리밍-zip 추출기(`wt_extract.py`, 자체테스트+fugu리뷰)로 4뷰 각 400프레임 추출 → 동일 스크립트로 n=200 재학습.

## 2. 핵심 발견 (가장 중요)
**데이터를 10× 늘리자 가설 판정이 뒤집혔다.**
- **HA 깊은특징: n=40 승(+0.021) → n=200 패(0.319 vs 0.436).** 더 깊은 net이 20에폭 고정에서 5× 데이터를 못 따라가 underfit. n=40의 "승리"는 소데이터 artifact였음 (robust하지 않음).
- **HB 증강: n=40 최악(0.355) → n=200 경쟁력(~0.407) + 최고 recall 코너(R0.860).** 데이터가 충분해지자 증강이 노이즈가 아니라 일반화로 작동 (이론대로).
→ **Cycle17의 "20프레임 판정은 데이터-regime artifact, 진짜 천장은 데이터양" 주장이 실증됨.** 단일 가설의 우열보다 **데이터 스케일이 결론을 지배**한다.

## 3. 사업·기술 함의
- n=200 baseline은 **고-precision 동작점이 매우 강함**(P0.858@thr0.10, P0.974@thr0.15) + 고-recall 코너(R0.827@thr0.02). 즉 데이터만 늘려도 **무-오경보 트래킹/상용분석(고precision)과 압사경보(고recall)** 양쪽 동작점이 실용 수준에 근접 → 비전공자 대시보드의 "위험" 색칠을 적은 오경보로 구동 가능.
- 아키텍처 미세조정(deep/aug/freeze)은 데이터가 적을 땐 노이즈, 충분할 땐 효과가 갈림 → **ROI 1순위는 여전히 데이터(프레임·뷰·다양성), 그다음이 모델**.

## 4. 다음 단계
1. deep를 **에폭↑(40~60)**로 재학습 → n=200에서 underfit 해소되는지(다시 baseline 추월하는지) 확인.
2. aug + 더 큰 데이터 조합 (현재 추세상 최고 recall 코너 → 압사경보 특화 동작점).
3. **운영 견고화(필수): 추출한 4뷰 프레임 서브셋을 Drive(MyDrive/WTx_full)에 영구 저장** → Colab 런타임 재활용(이번 사이클 3회 발생, /content 전체 소실) 시 재다운로드 없이 cp로 복구.
4. 그다음 학습형 BEV 동작점을 비전공자 대시보드에 연결(최종 목표).

## 부록 — 운영 교훈 (이번 사이클)
- EPFL zip은 **스트리밍 zip(local csize=0 + 데이터 디스크립터)** → csize 기반 추출 실패(WROTE 0). `zlib.decompressobj` 스트리밍 inflate로 해결(자체테스트+fugu리뷰 통과, WROTE 2444).
- **Colab 런타임이 idle 시 반복 재활용 → /content(다운로드·추출·clone) 전소.** active 셀은 유지되나 셀 사이 idle에서 소실. → Drive 영구화가 유일한 견고책.
- GLM 코드리뷰는 **계정 토큰 쿼터 소진**으로 빈 응답 → fugu로 대체(정상).
